## Epic Summary

Epic 1 delivered the technical foundation for the refreshed portfolio:

- Runtime and dependency baseline upgraded to the modern stack.
- Styling migrated from `styled-components` to Tailwind CSS.
- Docker runtime environment injection established through `docker-entrypoint.sh -> /dist/config.js -> window.__ENV -> src/config/env.ts`.
- GitHub Actions CI/CD workflow added for image build and publish.

All seven Epic 1 story files are marked `done` in `sprint-status.yaml`. This was the first retrospective for the project, so there was no previous retrospective to compare against.

## What Went Well

- The web UI continued to compile and render after fundamental code refreshes.
- The migration was handled incrementally rather than as a single risky cutover.
- Manual verification discipline stayed strong across the epic: build checks, dev-server checks, Docker checks, and browser validation were repeatedly used.
- Review findings were generally either fixed promptly or captured explicitly as deferred work rather than being lost.
- Story 1.6 and Story 1.7 preserved a coherent runtime configuration path instead of breaking environment handling during the CI/CD work.

## Challenges

- Visual fidelity drifted from the original landing-page composition. The current centered grid treatment reduced the edge-to-edge impact of the main page and background image.
- Live contact and integration behavior was not fully verified with real secrets and production-like configuration.
- Deferred work was tracked, but not consistently assigned to owners or target stories, creating uncertainty about when it will be addressed.
- The CI workflow is defined and locally reviewed, but it has not yet been confirmed through a real merge to `main`.
- Project guidance drift exists in `_bmad-output/project-context.md`, which still describes `styled-components` as the active styling system even though the implementation has moved to Tailwind.

## Key Decisions Made

- The landing-page visual drift will be handled inside Epic 2 Story 2.1, not through an Epic 2 replanning session.
- Live contact and integration verification is a critical-path task before Epic 2 starts.
- Deferred-item triage is also a critical-path task before Epic 2 starts.
- CI confirmation through a real `main` merge is important, but lower priority than contact and integration verification.
- The codebase is considered stable enough to proceed without a general hardening sprint.

## Takeaways

1. Incremental migration discipline preserved delivery confidence during a deep technical refresh.
2. Visual fidelity needs explicit protection during foundation work, or the UI can remain functional while drifting away from the intended experience.
3. Deferred work needs a scheduling lane, not just a tracking file.
4. Operational confidence requires real integration checks, not file inspection alone.
